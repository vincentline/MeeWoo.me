# Coder 技能优化建议文档

> 本文档记录了 AI Agent 在实际使用 Coder 技能过程中的体验反馈和优化建议，供后续技能迭代参考。

## 文档信息

| 项目       | 内容                   |
| -------- | -------------------- |
| **反馈来源** | AI Agent 实际使用体验      |
| **反馈日期** | 2026-03-07           |
| **测试场景** | 为 12 个 Python 脚本添加注释 |

## 一、整体评价

### 1.1 优点

| 优点       | 说明                           |
| -------- | ---------------------------- |
| ✅ 流程清晰   | 四步流程：查阅规则 → 编写代码 → 自检 → 提交建议 |
| ✅ 核心原则明确 | "不猜测，重实证" 避免编造代码             |
| ✅ 技能联动   | 与 integrity-check 技能有关联设计    |
| ✅ 规则优先   | 强制要求查阅规则文档                   |

### 1.2 不足

| 不足          | 影响程度           |
| ----------- | -------------- |
| ⚠️ 内容过于精简   | 高 - 实际操作缺乏细节指导 |
| ⚠️ 批量任务支持不足 | 高 - 无批量处理工作流   |
| ⚠️ 自检机制模糊   | 中 - 缺少具体检查项    |
| ⚠️ 规则查阅笼统   | 中 - 没有兜底策略     |

***

## 二、具体问题分析

### 2.1 缺少"任务复杂度判断"机制

**场景描述**：
需要为 12 个脚本添加注释，属于批量任务。

**问题表现**：

* 技能没有指导判断任务规模

* 没有建议使用 TodoWrite 工具

* Agent 需要自行判断是否创建任务追踪

**优化建议**：

```markdown
### 0. 任务评估 (新增步骤)

在开始编码前，先评估任务复杂度：

| 任务类型 | 判断标准 | 推荐行动 |
|---------|---------|---------|
| **简单任务** | 单文件、逻辑清晰 | 直接执行 |
| **中等任务** | 2-5 个文件、有关联 | 创建简单 TodoWrite |
| **复杂任务** | 5+ 文件、跨模块 | 详细规划 + TodoWrite + 分阶段执行 |
| **探索任务** | 目标不明确 | 先调研，再规划 |

**评估检查项**：
- [ ] 涉及多少文件？
- [ ] 是否跨多个模块？
- [ ] 是否需要修改核心逻辑？
- [ ] 是否有依赖关系？
```

***

### 2.2 "查阅规则"步骤过于笼统

**场景描述**：
为 Python 脚本添加注释时，不确定应遵循哪个规则文件。

**问题表现**：

* 只说"读取 index.md，定位领域"

* `.trae/rules/modules/` 下没有 Python 脚本规范

* 缺少兜底策略

**优化建议**：

```markdown
### 1. 查阅规则 (细化)

#### 1.1 规则定位流程

```

用户需求 → 识别关键词 → 匹配规则目录 → 读取规则文件
↓
无直接匹配？
↓
查找相似文件作为参考
↓
仍无参考？
↓
询问用户或使用通用规范

````

#### 1.2 规则匹配优先级

| 优先级 | 来源 | 示例 |
|-------|------|------|
| 1 | 直接匹配规则 | `modules/media/ffmpeg.ts.md` |
| 2 | 通用代码规范 | `core/coding-style.ts.md` |
| 3 | 相似文件参考 | `SearchCodebase` 查找同类文件 |
| 4 | 官方文档 | `WebSearch` 查阅官方文档 |

#### 1.3 兜底策略

当无法找到直接匹配的规则时：

1. **查找相似实现**：
   ```bash
   # 使用 SearchCodebase 搜索相似功能
   SearchCodebase("similar functionality")
````

1. **参考现有代码风格**：

   ```bash
   # 读取同类文件作为风格参考
   Read("path/to/similar/file.py")
   ```

2. **询问用户**：

   * 是否有特定规范需要遵循？

   * 是否有参考文件？

````

### 2.3 缺少"参考现有代码"的明确指导

**场景描述**：
添加注释时，不确定注释风格应该是什么样的。

**问题表现**：
- 技能没有明确说"参考现有代码风格"
- Agent 需要自己摸索

**优化建议**：

```markdown
### 2. 编写代码 (补充)

#### 2.1 风格一致性检查

在编写代码前，**必须**先了解现有代码风格：

**步骤**：
1. **定位参考文件**：
   - 同目录下的相似文件
   - 同类型的其他实现
   - 项目模板文件

2. **提取风格特征**：
   - 命名规范 (camelCase / snake_case)
   - 注释风格 (JSDoc / Google Style / NumPy Style)
   - 代码结构 (函数长度、模块组织)

3. **应用风格**：
   - 模仿现有代码的写法
   - 保持一致性

#### 2.2 风格参考示例

```python
# 如果项目中已有这样的注释风格：
def process_data(data):
    """
    处理数据。

    Args:
        data: 输入数据

    Returns:
        处理后的数据
    """
    pass

# 则新代码应保持相同风格，不要使用：
def process_data(data):
    # 处理数据
    # @param data 输入数据
    # @return 处理后的数据
    pass
````

````

### 2.4 "自检"步骤缺乏具体检查项

**场景描述**：
代码写完后，需要确认是否符合规范。

**问题表现**：
- 只说"静态检查"和"逻辑检查"
- 没有列出具体检查项
- 不知道项目是否有 lint 命令

**优化建议**：

```markdown
### 3. 自检 (细化)

#### 3.1 必检项清单

**代码质量**：
- [ ] 命名规范：变量/函数命名符合 `coding-style.ts.md`
- [ ] 类型安全：TypeScript 类型定义完整 / Python 类型注解
- [ ] 边界情况：空值处理、异常捕获
- [ ] 文档完整：函数有 docstring，复杂逻辑有注释

**业务逻辑**：
- [ ] 功能正确：核心功能测试通过
- [ ] 边界处理：极端输入不会崩溃
- [ ] 性能考虑：无明显性能问题

**项目规范**：
- [ ] 文件位置：放在正确的目录
- [ ] 导入规范：使用项目统一的导入方式
- [ ] 依赖检查：新增依赖已在 package.json 中

#### 3.2 自动化检查

**检测项目配置**：

| 配置文件 | 检查命令 | 说明 |
|---------|---------|------|
| `package.json` | `npm run lint` | ESLint 检查 |
| `package.json` | `npm run typecheck` | TypeScript 类型检查 |
| `pyproject.toml` | `ruff check .` | Python Lint |
| `pyproject.toml` | `mypy .` | Python 类型检查 |

**执行策略**：
1. 先检查项目是否配置了 lint/typecheck 命令
2. 若有配置，执行检查
3. 若无配置，跳过自动化检查，仅做手动检查
4. 检查结果有问题时，修复后再继续

#### 3.3 自检报告模板

````

## 自检报告

### 代码质量

* [x] 命名规范符合项目风格

* [x] 类型定义完整

* [x] 边界情况已处理

* [x] 文档注释完整

### 自动化检查

* [x] ESLint: 无错误

* [x] TypeScript: 无类型错误

### 发现问题

* 无

```
```

### 2.5 缺少"批量任务"工作流

**场景描述**：
需要处理 12 个文件的注释添加。

**问题表现**：

* 技能没有批量处理的指导

* 没有建议是否可以并行处理

* 没有进度追踪机制

**优化建议**：

```markdown
### 批量任务模式 (新增)

#### 适用场景
- 处理 3 个以上文件
- 相似性质的修改（如：添加注释、重命名、格式化）

#### 工作流程

```

┌─────────────────────────────────────────────────────────┐
│                    批量任务工作流                         │
├─────────────────────────────────────────────────────────┤
│  1. 评估阶段                                             │
│     - 统计文件数量                                       │
│     - 分析文件关联性                                     │
│     - 预估工作量                                         │
├─────────────────────────────────────────────────────────┤
│  2. 规划阶段                                             │
│     - 创建 TodoWrite 任务列表                            │
│     - 按优先级/依赖关系排序                              │
│     - 确定执行策略 (串行/并行)                           │
├─────────────────────────────────────────────────────────┤
│  3. 执行阶段                                             │
│     - 批量读取：一次读取多个文件                          │
│     - 逐个处理：每完成一个标记 completed                  │
│     - 定期同步：每 N 个文件汇报进度                       │
├─────────────────────────────────────────────────────────┤
│  4. 验收阶段                                             │
│     - 统一检查：确保风格一致性                            │
│     - 运行测试：验证功能正确                              │
│     - 总结汇报：列出所有变更                              │
└─────────────────────────────────────────────────────────┘

````

#### 批量读取策略

**一次读取多个文件**：
```python
# 推荐：批量读取
files = ["file1.py", "file2.py", "file3.py"]
for f in files:
    Read(f)  # 可以在一次响应中发起多个 Read

# 不推荐：逐个读取
Read("file1.py")
# 等待响应...
Read("file2.py")
# 等待响应...
````

#### 进度汇报模板

```
📊 批量任务进度

总文件数: 12
已完成: 8 (66.7%)
进行中: 1
待处理: 3

✅ 已完成:
- integrity-check/scripts/release.py
- integrity-check/scripts/scan_changes.py
- ...

🔄 进行中:
- knowledge-librarian/scripts/archiver.py

⏳ 待处理:
- webapp-testing/scripts/capture_errors.py
- webapp-testing/scripts/with_server.py
```

**场景描述**：

**问题表现**：

- 技能间的依赖关系不明确

**优化建议**：

### 4. 提交建议 (细化)

#### 4.1 技能协作决策树

````

代码变更完成
│
▼
┌─────────────────┐
│ 变更了核心逻辑？ │
└────────┬────────┘
│
┌────┴────┐
│         │
是        否
│         │
▼         ▼

测试通过？
│

│       │

│       │

#### 4.2 技能调用时机

| 触发条件 | 调用技能 | 说明 |
|---------|---------|------|
| 任何代码变更 | `integrity-check` | 检查 Inbox 覆盖率 |

```

└──→ webapp-testing (需要测试时)

````

## 三、新增功能建议

### 3.1 增加"快速参考卡片"

**目的**：避免每次都读完整规则，提高效率。

```markdown
## 快速参考卡片

### 常用操作速查

| 场景 | 行动 | 工具 |
|-----|------|------|
| 写新组件 | 参考现有组件 | `Read` / `SearchCodebase` |
| 改核心逻辑 | 先读规则 | `Read .trae/rules/modules/core/` |
| 不确定 API | 查文档 | `WebSearch` / `WebFetch` |
| 批量任务 | 创建追踪 | `TodoWrite` |
| 提交前检查 | 运行技能 | `/skill integrity-check` |

### 常见错误避免

| ❌ 不要 | ✅ 应该 |
|--------|--------|
| 猜测 API 用法 | 查文档或 SearchCodebase |
| 忽略现有代码风格 | 先读同类文件 |
| 批量任务不追踪 | 使用 TodoWrite |
| 直接写代码不查规则 | 先读 `.trae/rules/` |
| 提交前不检查 | 运行 integrity-check |

### 项目配置识别

执行前检查：
- `package.json` → 技术栈和脚本命令
- `.trae/rules/core/tech-stack.ts.md` → 项目技术栈
- `tsconfig.json` / `pyproject.toml` → 语言配置
````

### 3.2 增加"常见错误避免"章节

```markdown
## 常见错误避免

### 错误 1：不查阅规则直接写代码

**错误表现**：
```

用户: 帮我写一个新组件
Agent: \[直接开始写代码]

```

**正确做法**：
```

用户: 帮我写一个新组件
Agent: 让我先查阅项目规则和现有组件风格...
\[Read .trae/rules/core/coding-style.ts.md]
\[SearchCodebase "components"]
\[开始写代码]

```

### 错误 2：忽略现有代码风格

**错误表现**：
- 项目使用 camelCase，新代码用 snake_case
- 项目使用 JSDoc，新代码用 Google Style

**正确做法**：
- 写代码前，先读取同类文件
- 提取风格特征
- 保持一致性

### 错误 3：批量任务不追踪进度

**错误表现**：
- 处理 10 个文件，做到一半忘记进度
- 用户不知道当前状态

**正确做法**：
- 创建 TodoWrite
- 每完成一个标记 completed
- 定期汇报进度
```

### 测试任务

为 `.trae/skills` 目录下的 12 个 Python 脚本添加注释。

### 执行过程

1. **任务评估** (自行判断)
